# 폴더 구조 개선 제안

## 현재 폴더 구조의 문제점

현재 폴더 구조는 다음과 같은 문제점을 가지고 있습니다:

1. MVC 패턴이 혼재되어 있어 코드의 역할과 책임이 명확하지 않음
2. 기능별로 분리되어 있으나, 각 기능 내부에서 컨트롤러/서비스/엔티티가 혼합됨
3. 도메인 간의 관계가 명확하지 않음

## 제안하는 폴더 구조

NestJS 애플리케이션을 위한 더 나은 구조는 도메인 기반 접근법입니다. 각 도메인은 자체 모듈을 가지며, 그 안에 필요한 컴포넌트를 포함합니다.

```
src/
├── app.module.ts            # 애플리케이션의 루트 모듈
├── main.ts                  # 애플리케이션 진입점
├── common/                  # 공통 유틸리티 및 헬퍼
│   ├── constants/           # 상수 정의
│   ├── decorators/          # 커스텀 데코레이터
│   ├── filters/             # 전역 예외 필터
│   ├── guards/              # 전역 가드
│   ├── interceptors/        # 전역 인터셉터
│   ├── middleware/          # 전역 미들웨어
│   └── utils/               # 유틸리티 함수
├── config/                  # 환경 설정 관련 모듈
│   ├── config.module.ts
│   └── config.service.ts
├── database/                # 데이터베이스 설정 모듈
│   └── database.module.ts
├── health/                  # 헬스 체크 모듈
│   ├── health.module.ts
│   └── health.controller.ts
├── messages/                # 메시지 도메인 모듈
│   ├── dto/                 # 데이터 전송 객체
│   │   ├── create-message.dto.ts
│   │   └── update-message.dto.ts
│   ├── entities/            # 엔티티 정의
│   │   └── message.entity.ts
│   ├── messages.module.ts   # 메시지 모듈
│   ├── messages.service.ts  # 메시지 서비스
│   └── messages.controller.ts # 메시지 컨트롤러 (필요시)
└── webex/                   # Webex 도메인 모듈
    ├── bot/                 # Bot 관련 기능
    │   ├── bot.module.ts
    │   ├── bot.service.ts
    │   └── bot.controller.ts
    ├── commands/            # Webex 명령어 관련
    │   ├── commands.module.ts
    │   ├── commands.service.ts
    │   └── interfaces/
    │       └── command.interface.ts
    ├── interfaces/          # Webex 관련 인터페이스
    │   └── webex-types.ts
    ├── webex.module.ts      # Webex 루트 모듈
    └── webex-patch.ts       # Webex 라이브러리 패치
```

## 개선 사항

### 1. 도메인 기반 구조

각 비즈니스 도메인(messages, webex 등)은 자체 폴더에 모듈, 컨트롤러, 서비스, 엔티티 등 모든 관련 파일을 포함합니다. 이렇게 하면 관련 코드가 함께 위치하여 유지보수와 이해가 쉬워집니다.

### 2. 명확한 계층 분리

각 도메인 내에서:

- `entities/`: 데이터베이스 모델 정의
- `dto/`: 데이터 전송 객체 정의
- `*.service.ts`: 비즈니스 로직 처리
- `*.controller.ts`: HTTP 요청 처리
- `interfaces/`: 타입 정의

### 3. 공통 코드 분리

애플리케이션 전체에서 사용되는 공통 코드는 `common/` 폴더에 위치시켜 재사용성을 높입니다.

### 4. 구성 및 환경 설정 분리

`config/` 폴더에 환경 설정 관련 코드를 분리하여 애플리케이션의 설정을 중앙에서 관리합니다.

## 구현 계획

1. 새 폴더 구조 생성
2. 파일 이동 및 임포트 경로 수정
3. 모듈 간 종속성 업데이트
4. 테스트 및 확인

이 구조는 NestJS의 모듈 기반 아키텍처와 도메인 주도 설계(DDD) 원칙을 결합하여, 확장 가능하고 유지보수가 용이한 코드베이스를 제공합니다.
